1
Separate structured table data from text chunks
At ingestion, parse every detected table into SQLite rows, not text chunks. Key tables (hydrostatic, KN, GZ, tank capacity) need column-aware extraction. The 117 "table" chunks Docling found should become 117 SQL tables or rows, not 117 text blobs.

2
Replace BitNet 2B with a capable instruction model
BitNet 1.58-bit is a research model for edge inference, not a production reasoning model. For calculations and structured output, minimum viable is a properly quantized 7B model (Qwen2.5-7B-Q4_K_M, phi-4-mini-Q4_K_M, or Mistral-7B). Context must be ≥16k tokens.
3
Add a deterministic calculation engine — LLM never computes numbers
All arithmetic, interpolation, unit conversion, and compliance checks run as Python functions. LLM receives the trace output and writes prose around it. Block any LLM response containing numbers not found in the trace or source chunks.
def linear_interpolate(d, d1, d2, v1, v2):
frac = (d - d1) / (d2 - d1)
return round(v1 + frac * (v2 - v1), 1)
4
Fix the BM25 sparse index — it was dead at query time
Every query returned 0 BM25 results because the sparse index wasn't loaded. This means exact-match retrieval for terms like "8.17", "25551", "TPC" never fired. Fix the startup order — load FTS5 index before accepting queries, not concurrently with PDF ingestion.
5
SAMR must block, not warn, on score < 0.5
Currently SAMR appends a warning but still delivers a fabricated answer. A 0.38 faithfulness score means the answer is almost entirely hallucinated — it must be blocked and a "I cannot find that specific value in the document" message returned instead.
6
Add repetition detection to break infinite loops
Detect when the last N tokens repeat a pattern and stop generation immediately. A simple approach: if any 10-token window appears 3+ times in the generation, halt and return an error message. The "2500 tons" × 13 repetitions should have been caught after the 2nd occurrence.
7
Query intent classifier before any retrieval
Classify every query before routing: table_lookup / interpolate / unit_convert / compliance_check / explain / procedure. Each type routes to different handlers. "Displacement at 8.17m" is table_lookup+interpolate — it should never reach the LLM for the computation step.
8
Register a get_data tool (or rename lookup_table to match LLM expectation)
The LLM consistently tried to call get_data. Either register that tool name as an alias for lookup_table, or fine-tune the tool selection prompt to use the actual registered tool names. Unregistered tool calls should return a suggestion, not a silent error.
The fundamental architectural change needed: separate the retrieval plane (what data exists) from the calculation plane (what the numbers mean) from the generation plane (how to explain it).

The same pattern — tabular numerical data that requires domain-specific interpolation and calculation — appears across many fields. The architecture fix is identical in each case: extract structured data at ingestion, compute deterministically, let LLM explain.

Building / HVAC

Geotechnical
Soil classification tables, bearing capacity, pile load tables, groundwater interpolation
In every domain: the authoritative answer lives in a table, not a paragraph. The RAG pipeline must extract those tables into structured storage, not treat them as unstructured text. Domain-specific validators (e.g. "GZ at 0° must equal 0", "dosage must not exceed max listed in table") add an extra safety layer. The LLM's role is always the same: explain the result, never compute it.
